SOLID Principles

Introduction

SOLID is an acronym that stands for the following:

Single responsibility principle

Open/closed principle

Liskov substitution principle

Interface segregation principle

Dependency inversion principle

Single Responsibility Principle

This states that a class should have a single responsibility, but more than that, a class should only have one reason to change. Taking an example from Calculator program of the (simple) class called Calculator

According to the Single Responsibility Principle, each class should only have one job. If a class has many responsibilities, it is said to be coupled. When one responsibility is modified, the other responsibility is modified as well.

This class has its own property result, and it is retrieved by get() method.

Open-Closed Principle

Classes should be available for extension but closed for modification according to the open/closed paradigm. In other words, rather than being changed into something else, classes should be expanded to change functionality.

Lets take similar class calculator, all the mathematical methods follow OCP. Once we enter a method we complete the calculation and exit with a result

On the other hand if we had incorporated the division functionality then it would be kind of difficult to maintain the OCP as division by Zero needs exceptional handling.

Liskov Substitution Principle

The Liskov Subtitution Principle states that a client should be able to utilize any of a class's subtypes indistinguishably, without noticing, and thus without jeopardizing the intended behavior at runtime. Clients are entirely segregated and unaware of changes in the class hierarchy as a result of this.

Interface Segregation Principle

Make fine grained interfaces that are client specific Clients should not be forced to depend upon interfaces that they do not use. This principle deals with the disadvantages of implementing big interfaces.

To illustrate this completely, we will go with an example we implemented in the Calculator program under Calculation

Here we have implemented the Calculation class which is used as main class for Addition , Subtraction and the Multiplication classes.

Dependency Inversion Principle

Abstractions, not concretions, should be the source of dependency. High-level modules should not rely on low-level modules for their functionality. The abstractions used by both low and high level classes should be the same. Details should not be relied upon in abstracts. Abstractions should be used to determine the details.

Lets take similar Class calculator as above. The Addition function is getting the result from the get() method in addition Class which uses abstraction of the Calculation class.
